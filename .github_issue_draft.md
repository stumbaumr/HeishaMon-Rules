# Title

v4.1.4 regression: user-defined event calls (`name();`) inside `if … then … end` blocks no longer execute

# Body

## Summary

In firmware **v4.1.4**, calls to user-defined events (e.g. `dhw_start();`) are silently dropped when the call site is inside an `if … then … end` block. The condition is evaluated correctly, but the called event's body is never run. The same rule script works correctly on **v4.1.3**.

This is almost certainly a side effect of #891 / the rules.cpp changes between v4.1.3 and v4.1.4 (the only file changed in that release).

## Environment

- Hardware: ESP32 (`model-type-large`)
- Heat pump: Panasonic Aquarea
- Firmware (broken): **v4.1.4**
- Firmware (working): **v4.1.3**
- `force_rules`: enabled
- 14 rules loaded, `Rules active: 14`

## Minimal reproduction

```
on System#Boot then
   #flag = 0;
   #called = 0;
   setTimer(10, 10);
end

on timer=10 then
   setTimer(10, 30);
   if #flag == 0 then
      myevent();
   end
end

on myevent then
   #called = 1;
end
```

**Expected (v4.1.3 behaviour):** within ~10 s of boot, `#called == 1`.
**Observed on v4.1.4:** `#called` stays `0` indefinitely while `timer=10` fires every 30 s.

Calling the same user event at the top level of a handler *does* work in v4.1.4:

```
on @Main_Target_Temp then
   throttle();        ← this still works
end
```

So the regression is specifically about user-event invocations nested inside an `if` block.

## Real-world impact (what triggered this report)

A controller script uses a DHW-start sequence wrapped behind two `if` gates in `timer=10`:

```
if $hour == 13 && $minute == 0 && @DHW_Temp < ( @DHW_Target_Temp + @DHW_Heat_Delta ) && @Operating_Mode_State < 3 && #dhwStarted == 0 then
   dhw_start();
end
if @DHW_Temp < 18 && @Operating_Mode_State < 3 && #dhwStarted == 0 then
   dhw_start();
end
```

On v4.1.4 with `@DHW_Temp = 13`, `@Operating_Mode_State = 0`, `#dhwStarted = 0` — every gate satisfied — `dhw_start()` was never invoked, the heat pump stayed off, and the DHW tank kept dropping. Console evidence from `timer=10` after every 30 s fire:

```
==== timer=10 ====
timer #10 set to 30 seconds
rule #2 was executed in 7024 microseconds
>>> local variables
 0 $hour = 21
 1 $minute = 52
>>> global variables
 …
11 #dhwStarted = 0          ← would be 1 if dhw_start had run
```

No `==== timer=30 ====`, no `@SetHeatpump = 1`, and `#dhwStarted` never advances to `1` (the first statement inside `on dhw_start then` is `#dhwStarted = 1;`).

**Downgrading to v4.1.3 immediately resolved the issue** — DHW production started on the next `timer=10` fire after boot.

## Suspected cause

The only file changed between v4.1.3 and v4.1.4 is `HeishaMon/src/rules/rules.cpp` (PR #891, "Fixes some specific rules combination", addressing #885). The hunks touch `bc_assign_slots` (adding `continue` for `OP_CALL` when val==0 and for `OP_SETVAL`), `rule_create` TSEMICOLON handling (now calls `setval(node->a, 0)` after `OP_CLEAR`), and a `rule_run` condition change (`a != 4` gate, debug-check operator change). These touch the dispatch path for user-event invocations, which fits the observed symptom.

## Workaround

Inline the called event's body at every if-gated call site. For the script above, replacing both `dhw_start();` calls with the inlined body of `on dhw_start then … end` would restore behaviour on v4.1.4 — but at the cost of duplicating logic, so v4.1.3 is preferable until this is fixed.

## Ask

Please verify against the minimal repro above and, if confirmed, restore the v4.1.3 dispatch behaviour for `OP_CALL` inside conditional blocks. Happy to test a candidate fix on this device.
