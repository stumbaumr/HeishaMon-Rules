# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A single HeishaMon rules script (`rules.txt`) that automates a Panasonic Aquarea heat pump via the [HeishaMon](https://github.com/heishamon/HeishaMon) firmware. There is no build, test, or lint tooling — the file is uploaded to HeishaMon through its web UI (`/rules`), and the firmware's embedded rules engine parses and runs it. Validation happens at upload time on the device; iterate by editing locally, uploading, and watching live MQTT values / the HeishaMon log.

## Rules-engine language (essential to know before editing)

The script is written in HeishaMon's event-driven DSL. The syntax is defined here [rules](https://github.com/CurlyMoo/rules) — always get the current version before any editing!! Pay attention to the sigils, because they determine variable lifetime and source:

- `@Name` — **MQTT topic** exposed by HeishaMon. Read-only sensor topics (`@Outside_Temp`, `@DHW_Temp`, `@DHW_Target_Temp`, `@DHW_Heat_Delta`, `@DHW_Power_Consumption`, `@Heat_Power_Consumption`, `@Heatpump_State`, `@Operating_Mode_State`, `@ThreeWay_Valve_State`, `@Defrosting_State`, `@Main_Outlet_Temp`, `@Main_Inlet_Temp`, `@Main_Target_Temp`, `@Room_Thermostat_Temp`, `@Max_Pump_Duty`, `@Quiet_Mode_Level`, `@Z1_Heat_Request_Temp`, `@Z1_Cool_Request_Temp`, `@Z1_Heat_Curve_Target_Low_Temp`, `@Z1_Heat_Curve_Target_High_Temp`, …) and write-capable command topics (`@SetHeatpump`, `@SetOperationMode`, `@SetPump`, `@SetMaxPumpDuty`, `@SetDHWTemp`, `@SetCurves`, `@SetZ1HeatRequestTemperature`, `@SetZ1CoolRequestTemperature`, `@SetQuietMode`, `@SetForceDefrost`). Assigning to a `@Set…` topic publishes a command to the heat pump.
- `#name` — **persistent global** variable, survives across rule firings (and, with HeishaMon's flash persistence, across reboots).
- `$name` — **local** variable, scoped to a single `on … end` block.
- `%name` — **system** variable supplied by the engine (`%hour`, `%minute`).

Events / blocks:

- `on System#Boot then … end` — runs once at firmware start. Used here to seed all `#…` defaults and arm the periodic timer.
- `on timer=<id> then … end` — fires when a timer set by `setTimer(<id>, <seconds>)` elapses. Timer `10` is rearmed each tick to act as a 30-second clock; timers `21`, `22`, `30`, `31`, `32` are one-shot phases of the DHW/recovery state machine.
- `on @<Topic> then … end` — fires when that MQTT topic's value changes.
- `on <name> then … end` — user-defined event, invoked by calling `<name>()` from another block (used here for `throttle` and `dhw_start`).

Built-ins used: `setTimer(id, seconds)`, `concat(…)` (string concat — used to build the JSON payload for `@SetCurves`), `round()`, `floor()`. The `%` operator is integer modulo.

A subtle parity convention to keep in mind when reading: **`@Operating_Mode_State` even == heating, odd == cooling**, and `+ 4` toggles the DHW-active variant — that's why the code does `@Operating_Mode_State % 2 == 0` to branch heat vs. cool and `#OpModeBeforeDHW + 4` to enter DHW mode while preserving the underlying heat/cool selection.

## What the rules actually do (high-level)

The script is one coordinated controller — editing one block in isolation will usually break another. The pieces:

1. **Boot (`System#Boot`)** — seeds `#…` state, disables the heat pump, sets pump-duty caps (`#maxPumpDutyQuiet = 112`, `#maxPumpDutyPower = 170`), and arms the master timer (id `10`) with a 40 s first tick (it re-arms itself at 30 s thereafter).

2. **Master clock (`timer=10`)** — the central scheduler. Re-arms itself every 30 s, then (gated by `@Operating_Mode_State < 3 && #dhwStarted == 0`) **continuously snapshots the pre-DHW state** into `#stateBeforeDHW` / `#OpModeBeforeDHW`, so even a DHW run started from the Panasonic remote button (valve flip with no `dhw_start`) restores the right on/off state afterwards. The `#dhwStarted == 0` half of that guard is essential: `dhw_start` turns the unit on (`@SetHeatpump = 1`) well before op mode flips to DHW (up to 540 s in cold weather), so without it the snapshot would be clobbered to `1` mid-startup and a unit that was **paused** (`#offAtNight == 2`) would be turned back on by the `timer=32` recovery instead of returning to off. After the snapshot it checks `%hour`/`%minute` for time-of-day actions:
   - **08:00** — if `@Outside_Temp < 12`, raise pump duty to the "power" cap; if the unit was paused overnight in heating mode (`@Operating_Mode_State == 0 && #offAtNight == 2 && @Heatpump_State == 0`), restart it; clear the nighttime quiet floor (`#QuietMinimumValue = 0`).
   - **09:00** — fallback restart for the cooling case: when paused (`#offAtNight == 2`, `@Heatpump_State == 0`) in cooling mode (`@Operating_Mode_State == 1`) and it's hot (`@Outside_Temp > 27`), turn the unit back on.
   - **13:00** (when `@DHW_Temp < @DHW_Target_Temp + @DHW_Heat_Delta` and not already in DHW) **and anytime `@DHW_Temp < 20`** — trigger the DHW heat-up via `dhw_start()` (both paths gated by `@Operating_Mode_State < 3` and `#dhwStarted == 0`).
   - **18:00** — decide whether to pause overnight (only if `#offAtNight == 1 && @Heatpump_State == 1`): in heating mode, mark `#offAtNight = 2` only if the room is already warm and it's mild (`@Room_Thermostat_Temp > 21 && @Outside_Temp > 8`); in cooling mode, set `#offAtNight = 2` and turn the unit off immediately.
   - **19:00** — drop pump duty back to the quiet cap; in heating mode finish the off-at-night transition (turn off if `#offAtNight == 2 && @Heatpump_State == 1`); raise the quiet-mode floor (`#QuietMinimumValue = #QuietAtNight`).

3. **DHW cycle** — driven by the three-way valve state and a small state machine. The goal is to keep the compressor running continously - even when switching back to heating mode. This is achieved by adjusting the heating curve for a smooth ramp down. **Command-rate budget:** to avoid overrunning HeishaMon's heat-pump command throughput, every phase of this machine is structured to emit **at most two `@Set…` commands per firing** — bursts that used to fire inline are split across staggered one-shot timers (see the timer roles below). Preserve this when editing: if a block would emit a third command, push it onto a new timer phase.
   - `on @ThreeWay_Valve_State`: when the valve flips to DHW (`== 1`), the rules **compute a dynamic DHW target** by linearly interpolating between 43 °C (at −5 °C outside) and 50 °C (at 30 °C outside) — clamped at the endpoints — and write it via `@SetDHWTemp`, then **snapshot** the current curve targets into `#targetLowBeforeDHW` / `#targetHighBeforeDHW` (from `@Z1_Heat_Curve_Target_{Low,High}_Temp`) and write temporarily-narrow curves (both high and low set to `@DHW_Target_Temp + 6`, via `$targetAfterDHW`) so after leaving DHW mode the compressor does not immediatley turn off — that is the two inline commands. It then arms `timer=22` (1 s, re-asserts DHW op mode) and `timer=21` (31 s, snapshots the pump-duty cap into `#maxPumpDutyBeforeDHW`, raises duty to the power cap, and turns the waterpump to test mode `@SetPump = 1`). When the valve flips back (`== 0`) and no defrost is in progress (`#DefrostWhileDHW == 1`), it arms the three-phase recovery chain (`timer=30`/`31`/`32`). Using the valve state allows to still use the DHW button on the Panasonic Remote Control. During DHW production the waterpump is set to test mode to achieve the highest volume flow possible.
   - `on @Defrosting_State`: if a defrost starts *during* DHW (`@Defrosting_State == 1 && @ThreeWay_Valve_State == 1`), mark `#DefrostWhileDHW = 2`, set op mode to 0, then arm `timer=30` (30 s, heat request 5 + restore pre-DHW curves) and `timer=31` (60 s, restore the pump-duty cap + `@SetPump = 0`). When the defrost ends (`#DefrostWhileDHW == 2`), set `#DefrostWhileDHW = 1` and return to DHW mode (`#OpModeBeforeDHW + 4`). Not restoring the pre-DHW curves will make us loose the curves after Defrost. The goal is to use the energy from the heating buffer instead of the energy of the DHW tank.
   - `on @DHW_Power_Consumption`: when power draw drops to 0 while in DHW mode (`@Operating_Mode_State >= 3`) and the target is reached (`@DHW_Temp >= @DHW_Target_Temp || @Main_Outlet_Temp >= @Main_Target_Temp`), arm `timer=32` (1 s) to restore the pre-DHW op mode and on/off state. This flips the three-way valve back, which is what actually arms the recovery chain in `on @ThreeWay_Valve_State`.
   - `dhw_start` + `timer=22`: cold-weather (`@Outside_Temp < 5`) DHW runs are preceded by a forced defrost (`@SetForceDefrost = 1`) which takes well below 9 minutes and therefore a 540 s wait; warm-weather DHW runs arm `timer=22` after 35 s. `dhw_start` sets the `#dhwStarted` guard and turns the unit on (`@SetHeatpump = 1`) — it no longer snapshots the pre-DHW state itself; that snapshot now lives in `timer=10` (see the master-clock note above, which is why the `#dhwStarted` guard there matters). `timer=22` enters DHW op mode (`#OpModeBeforeDHW + 4`) and clears the `#dhwStarted` re-entry guard. The subsequent valve flip then drives the `@ThreeWay_Valve_State == 1` path above.
   - **Post-DHW recovery chain** (`timer=30`/`31`/`32`, armed when the valve flips back to heating): `timer=30` (240 s) sets heat request to 5 and restores the pre-DHW curves; `timer=31` (270 s) restores the pump-duty cap and turns off the waterpump test mode (`@SetPump = 0`); `timer=32` (300 s) re-asserts the pre-DHW op mode and heatpump state. The staggered timing gives a slow ramp-down of the heating curve, and keeps each phase within the two-command budget.

4. **Adaptive request temperature (`throttle`)** — fired (via `throttle()`) whenever `@Main_Target_Temp`, `@Main_Outlet_Temp`, or `@Main_Inlet_Temp` changes. Takes the heating branch when in a heating op mode **or** while the valve is in DHW (`@Operating_Mode_State % 2 == 0 || @ThreeWay_Valve_State == 1`), otherwise the cooling branch. Computes an outlet-vs-target delta, optionally biased by `#roomDelta` when it's cold and the room is already warm (`@Outside_Temp < 7 && @Room_Thermostat_Temp > 22`), then sets `@SetZ1HeatRequestTemperature` (capped at 5) or `@SetZ1CoolRequestTemperature` (clamped to −2…3) to nudge the curve up or down. Also sets `@SetQuietMode` based on how hard the unit is working — small delta → quiet floor (`#QuietMinimumValue`), large delta → quiet level 3. The goal is to keep the compressor running as long as possible for a high lifetime of the compressor. When the house does not take the produced energy the quiet mode reduces the energy output of the compressor.

5. **Smart pause (`on @Heat_Power_Consumption`)** — if the unit is running but drawing 0 W in mild weather (`@Outside_Temp > 10`) with a positive heat request, pause it (`#offAtNight = 2`, `@SetHeatpump = 0`). The morning 08:00 / 09:00 logic in `timer=10` is what un-pauses it. The goal is again to keep the compressor off untill the next day if the produced energy is not consumed anymore.   

## Editing guidance

- **Tunables live at the top of `System#Boot`.** Things like `#maxPumpDutyQuiet`, `#maxPumpDutyPower`, `#QuietAtNight`, and the curve seeds are the intended knobs — change those rather than rewriting downstream blocks.
- **DHW target curve** (the 43→50 °C interpolation) is inlined in `on @ThreeWay_Valve_State`; the four boundary constants (`$minDHWTarget`, `$minOutside`, `$maxDHWTarget`, `$maxOutside`) are local `$` vars set on each valve transition. The narrow holding curve written for that transition is `@DHW_Target_Temp + 6` (both high and low), via the local `$targetAfterDHW`.
- **Don't reuse timer IDs.** `10`, `21`, `22`, `30`, `31`, `32` are taken and carry distinct semantics; pick a new ID for new timers.
- **Two-command-per-firing budget.** No single rule firing should emit more than two `@Set…` commands (HeishaMon command-throughput limit). The DHW/recovery state machine deliberately splits bursts across staggered timers to honor this — when adding behavior, prefer arming a new timer phase over appending a third command to an existing block.
- **Pause guards.** Many blocks gate on `@Heatpump_State == 1` so they don't fight a manually-stopped unit. Preserve that pattern.
- **The `#dhwStarted` flag** prevents double-firing of `dhw_start` from the two trigger paths in `timer=10` (the 13:00 schedule and the low-tank fallback). Don't remove the guard without replacing it.
- **`@SetCurves` payload** is built with `concat(…)` into a JSON-ish string the firmware parses — match the exact shape `{zone1:{heat:{target:{high:<n>,low:<n>}}}}` when editing.
- **Firmware floor for `name();` inside `if`-bodies.** v4.1.4 and v4.1.5 miscompile two adjacent `if … then <call>; end` blocks (issue [#894](https://github.com/heishamon/HeishaMon/issues/894)). The `dhw_start` call sites in `timer=10` rely on the fix in [IgorYbema/HeishaMon@913e8eb](https://github.com/IgorYbema/HeishaMon/commit/913e8ebcf5b5f6cc29b2d83b5562af1704dc506f) (reset `mathcnt` when a call statement closes an if-block). Do not flash a firmware older than the first heishamon release containing that commit — if forced to, inline the `dhw_start` body at both call sites and delete the `on dhw_start` rule.
